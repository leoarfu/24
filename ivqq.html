02 Context — What OneSat Is

OneSat is Airbus DS's fully software-defined GEO communications satellite platform. Its differentiator is in-orbit reconfigurability: beam coverage, bandwidth allocation, and frequency plans can be changed post-launch via software upload, without hardware modification. Customers include Intelsat (Galaxy 35/36) and Eutelsat.

The Ground Segment is correspondingly more complex than a fixed GEO ground system: dynamic payload configuration, TT&C for a software-defined payload, service management across reconfigurable beams, and interfaces between the Network Management System, Payload Control System, and Mission Control System. The IVVQ Manager owns proving correctness of this system end-to-end, from factory through site.

CCSDS / TT&C Protocol Basics
If asked: "How does the ground system command and monitor a software-defined payload?"
CCSDS (Consultative Committee for Space Data Systems) defines the standard frame structures for telemetry and telecommand. SLE (Space Link Extension) is the protocol layer between the ground station and the mission control system for forwarding TM/TC over IP. For OneSat specifically, the Payload Control System sits between the Mission Control System and the payload — configuration commands for beam/frequency reconfiguration flow through this chain and need their own verification, separate from classic bus TT&C. You don't need deep protocol fluency, but you should be able to say this much without hesitating.
V-Model and IVVQ Phasing
If asked to structure a validation approach from scratch.
Requirements → design → implementation on the left side; unit test → integration test → system test → acceptance test climbing back up the right side, each test phase verifying against the corresponding left-side definition phase. For ground segment: subsystem integration (e.g., NMS alone) → system integration (NMS+PCS+MCS together) → Factory Integration Test (full chain, controlled environment) → Qualification (formal proof against requirements) → Site Acceptance Test (customer environment, contractual milestone). Be ready to sketch this on a whiteboard without notes.
CCB Mechanics
If asked how you'd handle a change request mid-campaign.
Change Control Board: any proposed change (requirement clarification, design fix, anomaly correction) gets classified by impact before being accepted — does it require no re-test, partial regression, or full regression of the affected test suite. The IVVQ role's job in CCB is to give a fast, defensible impact assessment, not veto power. You haven't sat in one, but you have made equivalent calls informally when deciding whether a provider-side network change required re-running onboarding validation at HBC+ — that's a usable bridge, stated honestly as informal/analogous.
Link Budgets & C/N — you have this, just refresh the numbers
If asked a quick technical sanity-check question on RF performance.
This is genuinely your strength from iDirect — don't let nerves make you hedge on it. Carrier-to-noise ratio, Eb/N0, modulation/coding trade-offs (DVB-S2 ACM), rain fade margin. Close by stating what you bring that a pure IVVQ-career candidate might not: direct experience with how systems actually fail in the field, not just how they're specified to behave.Signal keywords: fault isolation · requirements traceability · honest gap · ramp plan
OneSat's payload is software-defined — coverage and frequency plans change via uplink. How would you approach validating that?
The validation space grows combinatorially with configuration options — you can't test every permutation, so the strategy needs configuration equivalence classes, validating representative cases rather than the full space. Every uplinked reconfiguration needs a regression suite, not just a one-time qualification. You'd also need to validate the configuration management process itself — not just that a given end-state is correct, but that the path to get there is safe. This is reasoning you can do from system-architecture instinct even without IVVQ tooling experience — say so.Signal keywords: configuration equivalence class · regression after uplink · process validation, not just state validation
As Chief Engineering, I'm accountable for IVV outcomes on this program. What ramp-up support would you need from my organization in your first 90 days?
Answer this concretely, not vaguely — she's asking because she'll personally bear the cost. Name specifics: shadowing an active FIT or qualification campaign in the first month, a structured walkthrough of the current test management tooling and traceability process, time with the outgoing or adjacent IVVQ leads to absorb tribal knowledge of OneSat-specific gotchas, and an explicit understanding of what decisions you can make autonomously vs. what needs her sign-off early on. A specific, bounded ask is more reassuring than "I'll figure it out."Signal keywords: bounded ramp plan · explicit decision authority · shadow before lead
Describe a time you found a serious fault in a live system. What did you do?
Pull a real iDirect story — RF/baseband/IP fault isolation under customer pressure. Structure: symptom as reported by customer → your diagnostic path across the stack (RF → modem → IP) → root cause → fix → what you'd change about the system's test coverage in hindsight to have caught it earlier. That last beat is the bridge to IVVQ thinking: turning an operational failure into a validation-process lesson.Signal keywords: cross-layer diagnosis · root cause · coverage gap in hindsight
What's the difference between Factory Integration Test and Site Acceptance Test, and why does it matter?
FIT: controlled environment, full instrumentation access, no customer pressure. SAT: customer's environment, the ground station's existing infrastructure (antenna, SSPA, modem chain) now interacts with your delivered system, real RF environment including interference, contractual milestone pressure. The goal of FIT is to make SAT a confirmation, not a discovery exercise — every issue found at site is more expensive and more visible than the same issue found in factory. You know this conceptually from iDirect's customer deployment context even without having run a formal FIT/SAT pair yourself.Signal keywords: FIT · SAT · delta environment · discovery vs confirmation
